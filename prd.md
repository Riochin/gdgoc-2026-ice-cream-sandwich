## 概要

ユーザーの自然言語の要望（例：「渋谷で静かなカフェ、電源あり」）をチャットUIから入力すると、検索エージェントが要件を解釈してレストラン候補を検索・絞り込みし、候補一覧と詳細（評価、住所、営業時間、リンク等）を提示する。

## 背景 / 課題

- 既存の検索UIは「地域/ジャンル/価格帯」などのフォーム入力が多く、ユーザーの意図（雰囲気、用途、同伴者、こだわり）を表現しづらい
- 検索→条件調整→再検索の反復が発生しやすい
- チャット形式で要望を伝えるだけで、条件の不足をエージェントが質問し補完しながら最適化できる体験を提供したい

## 目的 / ゴール

- 自然言語での検索入力から、ユーザー意図に沿ったレストラン候補を提示できる
- 条件が不足している場合、エージェントが追加質問して要件を確定できる
- 検索結果の説明（なぜその候補か）と、比較しやすい表示ができる

## スコープ

### In scope

- チャットウィンドウUI（会話ログ、入力欄、送信、ローディング、エラー表示）
- 検索エージェント（要件抽出、検索クエリ生成、検索実行、結果整形、追加質問）
- レストラン情報表示（一覧/詳細：評価、住所、営業時間、写真（あれば）、リンク）
- 条件フィルタ（価格帯、距離、ジャンル、営業時間など）の解釈と適用（自然言語由来）

### Out of scope（初期）

- 予約機能（外部連携）
- ユーザーアカウント/ログイン、履歴の永続保存（MVPでは任意）
- 高度なパーソナライズ（嗜好学習）

## 想定ユーザー

- 「近くで良い店をサクッと探したい」一般ユーザー
- 目的ベース（デート/作業/家族/接待など）で店を探すユーザー
- 条件を文章で伝えたいユーザー（フォーム入力が面倒）

## ユースケース（代表例）

1. 「新宿で20時以降、1人で入れて、静かなラーメン屋」
2. 「恵比寿でデート向け、予算は2人で1.5万円くらい」
3. 「今いる場所の近くで、ベビーカーOKのカフェ」
4. 「渋谷で電源とWi-Fiがあって2時間作業できる」

## 体験設計（会話フロー）

### 基本フロー

1. ユーザーが要望を文章で入力
2. エージェントが要件を抽出（エリア/目的/人数/予算/料理/制約など）
3. 不足情報があれば確認質問（最大1〜2問を目安）
4. 検索実行 → 候補を3〜10件提示（理由付き）
5. ユーザーが追加条件（「もっと安く」「駅から近く」等）を追記 → 再検索/再ランキング

### 追加質問の例

- エリアが曖昧：「今いる場所周辺ですか？それとも◯◯駅周辺ですか？」
- 予算がない：「予算感はどのくらいですか？（例：〜1,000円 / 〜3,000円 / 〜6,000円）」
- 人数：「人数は何名ですか？」

## 機能要件

### 1) チャットUI

- 会話履歴（ユーザー/エージェントの吹き出し）
- 送信中のインジケータ
- 途中で再入力できる（入力欄は常に利用可能）
- エラー時のリトライ導線

### 2) 要件抽出（NLU）

- 入力文から以下を抽出（可能な範囲で）
    - エリア（駅名、区、市、現在地）
    - ジャンル（和食/イタリアン/カフェ等）
    - 目的（作業/デート/会食/家族/一人/団体等）
    - 予算（上限/下限、ランチ/ディナー）
    - 制約（禁煙、個室、電源、Wi-Fi、バリアフリー、子連れ等）
    - 時間（今/今日/◯時以降、曜日）
- 不確実性（推定）と「確認が必要か」を判断できる

### 2.1) 会話エージェント実装（ADK）

- 会話側AI（要件抽出、質問生成、検索クエリ生成、結果要約）はエージェントSDK（ADK）で実装する
- 会話コンテキスト（確定済み条件/未確定条件/前回の検索結果）をADKのセッション状態として保持する
- ツール呼び出し（例：searchRestaurants、getRestaurantDetail、fetchMenuFromWebsite）はADKのツールとして定義し、エージェントが必要に応じて選択・実行する
- 失敗時（ツール失敗・タイムアウト・結果0件）はADK側でフォールバック方針（再試行/条件緩和提案/追加質問）を統一的に扱えるようにする

### 3) 検索・ランキング

- 外部APIやDBからレストラン候補を取得
- フィルタ条件を適用
- ランキング（距離、評価、レビュー数、要望一致度などの重み付け）
- 結果が少ない場合の提案（条件緩和の提案）

### 4) 結果表示

- 一覧：店名、評価、距離（任意）、価格帯、主要タグ、短い推薦理由
- 詳細：住所、営業時間、電話、Webリンク、地図リンク、写真（あれば）
- 「この店を選んだ理由」を最低1行で表示

### 5) 追加入力による再検索

- ユーザーの追加要望を既存要件にマージ
- 会話コンテキストを保持（前提条件を覚える）

## 非機能要件

- パフォーマンス：初回回答までの目標 3〜5秒（検索API次第、段階的に表示可）
- 可用性：検索失敗時に理由表示 + 代替案（条件緩和/再試行）
- セキュリティ：APIキーの秘匿、スクレイピング（ボーナス）時のアクセス制御
- ログ：会話ログと検索クエリを観測できる（デバッグ用）

## データ要件

- レストラン基本情報（店名、住所、緯度経度、カテゴリ、営業時間、評価、レビュー数、URL）
- 取得元候補：Google Places / Hotpepper / 食べログ等（実装都合で選定）
- キャッシュ：同一クエリの短期キャッシュ（レート制限対策）

## 技術調査メモ：ADK / MCP（実装方針）

### ADK（Agent Development Kit）でやること

- 会話エージェント（オーケストレーション）をADKで実装し、ツール呼び出し中心の構成にする
- 代表的な構成要素（例）
    - Agent: ユーザー入力→要件抽出→不足確認→ツール実行→要約
    - Tools: 外部API/DB/スクレイピングなどの副作用を伴う処理
    - Session / Memory: 会話状態（確定条件・未確定条件・候補一覧・ユーザーの優先度）を保持
- 開発手順（MVP）
    - 最小のツール（searchRestaurants / getRestaurantDetail）から開始
    - 失敗時の方針（0件/タイムアウト/レート制限）を共通化してUXを安定させる
    - ログ（プロンプト/ツール入出力/選択理由）を保存してチューニング可能にする

### MCP（Model Context Protocol）でやること

- エージェントから外部機能を呼ぶ「ツール公開」を標準化するためにMCPを利用する
- 構成（推奨）
    - ADK（MCP Client）: エージェント側。必要なツールを呼び出す
    - MCP Server: ツール提供側（検索API、メニュー取得、DBアクセスなど）
- このPJでのMCP Server候補
    - restaurant-search server
        - searchRestaurants(query, filters)
        - getRestaurantDetail(restaurantId)
    - menu-fetch server（ボーナス）
        - discoverMenuUrl(restaurantUrl)
        - fetchMenuFromWebsite(menuUrl)
        - parseMenu(rawHtmlOrPdf) → 構造化（カテゴリ/品名/価格/注記）
- 通信方式は要件とインフラに合わせて選定（例：SSE / Streamable HTTP）
- メリット
    - ツールを別プロセス/別言語で独立実装でき、会話エージェントの更新頻度と分離できる
    - 将来、別クライアント（別エージェント、別アプリ）から同じツール群を再利用できる

## 画面案（ラフ）

- 画面は大きく「チャット」+「検索結果パネル（カード一覧）」の2ペイン
- モバイルではタブ切り替え（チャット / 結果）

## 成功指標（例）

- 検索→候補提示までの平均ターン数（少ないほど良い）
- 検索結果クリック率（外部リンク/地図リンク）
- 条件再入力率（高すぎると要件抽出が弱い可能性）
- ユーザー満足（簡易アンケート）

## リスク / 検討事項

- 自然言語の曖昧さ（地名の同名、目的の解釈）
- 外部APIのレート制限、費用
- 住所/営業時間の鮮度
- ランキングの透明性（理由の説明）

## ボーナス課題：レストランのWebサイトからメニュー情報を取得して表示

### 目的

- 公式サイト等にしかないメニュー情報を補完し、意思決定を早める

### 仕様（案）

- 詳細画面に「メニュー」セクションを追加
- 情報源：レストラン公式サイトのメニュー/お品書きページ（URLが分かる場合）
- 取得方法：
    - 可能なら構造化データ（JSON-LD / OpenGraph / メニューPDF）を優先
    - ない場合はHTMLから抽出（スクレイピング）し、LLMで整形する
- 表示：カテゴリ（料理/ドリンク）、代表メニュー名、価格、注記（税/サービス料など）
- 取得できない場合：理由表示（「サイト構造上取得不可」等）＋代替（外部メニューサービスへのリンク）

### 非機能・注意点

- 利用規約/robots.txtの遵守（禁止サイトは除外）
- アクセス頻度制限（キャッシュ、バックオフ）
- 取得結果の鮮度（更新日時が不明な場合は注意書き）
